To answer your direct question: **No, using "base classes" like `ProcBase` and `DecoderBase` is not idiomatic (kosher) in Go.** Go intentionally omits inheritance. The pattern used in your code—storing a named `Base` struct and manually writing boilerplate methods to forward calls to it—is a classic artifact of translating C++ directly to Go. While it works functionally, it creates tight coupling, monolithic dependencies, and a lot of unnecessary code. Go strongly prefers **composition and interfaces** over inheritance. 

Here is a step-by-step incremental refactoring plan to rid your codebase of these C++-isms and make it idiomatic Go, moving from easy wins to deeper architectural changes.

### Step 5: Clean Up Error Handling
The codebase mixes standard Go errors (`error`) with a custom C++ style `ocsd.DatapathResp` enum. 
Furthermore, components call `d.Base.LogError(...)` internally instead of letting the caller handle it.

**The Go Way:**
1. Let the `error` bubble up to the top boundary of the application using `fmt.Errorf("%w: ...", err)` (which you are partially doing, which is great). 
2. Stop logging errors deep inside the decoders. Return the error, and let the top-level loop (e.g., `listTracePackets`) log the error exactly once. This removes the need for `DecoderBase` to carry around a logger entirely.
