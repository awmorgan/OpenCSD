To answer your direct question: **No, using "base classes" like `ProcBase` and `DecoderBase` is not idiomatic (kosher) in Go.** Go intentionally omits inheritance. The pattern used in your code—storing a named `Base` struct and manually writing boilerplate methods to forward calls to it—is a classic artifact of translating C++ directly to Go. While it works functionally, it creates tight coupling, monolithic dependencies, and a lot of unnecessary code. Go strongly prefers **composition and interfaces** over inheritance. 

Here is a step-by-step incremental refactoring plan to rid your codebase of these C++-isms and make it idiomatic Go, moving from easy wins to deeper architectural changes.

### Step 2: Eliminate Two-Phase Initialization (`Init()` methods)
C++ code often relies on empty constructors followed by an `Init()` method because of how C++ handles virtual tables and memory allocation. 
```go
// Current C++-style approach
d.Base.Init(fmt.Sprintf("%s_%d", "DCD_ETMV3", instID), logger)
```
**The Go Way:** Go prefers factory functions (`New...`) that return fully initialized, ready-to-use structs. You should remove the `Init()` methods on the base structs and initialize their fields directly inside your `NewPktDecode` and `NewPktProc` functions.
```go
func NewPktDecode(cfg *Config, logger ocsd.Logger) *PktDecode {
    return &PktDecode{
        DecoderBase: common.DecoderBase{
            Name:         fmt.Sprintf("DCD_ETMV3_%d", instID),
            Logger:       logger,
            ErrVerbosity: ocsd.ErrSevNone,
            // ... other initializations
        },
        // ...
    }
}
```

### Step 3: Break Down the Monolithic Base Classes (Composition)
`DecoderBase` and `ProcBase` are "God objects." They know about Logging, Error Verbosity, Operational Flags, Memory Access, and Instruction Decoding. In Go, you should separate these concerns into smaller, injected dependencies.

**The Go Way:**
1. **Logging:** Instead of baking `LogError` and `LogMessage` into a base class, pass the `ocsd.Logger` directly to the structs that need it, or create a small `type Logger struct` that can be embedded.
2. **Flags/State:** Extract `OpFlags` and `SupportedOpFlags` into a dedicated `type OpMode struct` that can be embedded.
3. **Interfaces:** Instead of requiring a base struct, define small interfaces for what these components *do*. 
```go
// Define small interfaces
type OpModeManager interface {
    ComponentOpMode() uint32
    SupportedOpModes() uint32
}

type TraceElementOutputter interface {
    OutputTraceElement(traceID uint8, elem *ocsd.TraceElement) ocsd.DatapathResp
}
```

### Step 4: Refactor C++-Style State Machines
In `etmv3/decoder.go`, functions like `ProcessPacket()` use a very C++-style `while(!pktDone)` loop wrapped around a massive `switch(currState)` block, mutating shared state variables.
```go
// Current C++-style approach
for !pktDone {
    switch d.currState {
        case noSync:
            resp = d.sendUnsyncPacket()
            d.currState = waitAsync
// ...
```
**The Go Way:** 1. **Return early:** Break the switch cases into distinct handler functions that return the next state and the response. 
2. **Minimize mutable state:** Instead of mutating `d.currState` heavily inside deep switch statements, transition states by returning them: `nextState, resp := d.handleNoSync()`.

### Step 5: Clean Up Error Handling
The codebase mixes standard Go errors (`error`) with a custom C++ style `ocsd.DatapathResp` enum. 
Furthermore, components call `d.Base.LogError(...)` internally instead of letting the caller handle it.

**The Go Way:**
1. Let the `error` bubble up to the top boundary of the application using `fmt.Errorf("%w: ...", err)` (which you are partially doing, which is great). 
2. Stop logging errors deep inside the decoders. Return the error, and let the top-level loop (e.g., `listTracePackets`) log the error exactly once. This removes the need for `DecoderBase` to carry around a logger entirely.
